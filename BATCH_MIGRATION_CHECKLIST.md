# Batch-Based Student Identity Migration Checklist

Goal: introduce cohort-safe student identity using roll format `YYDDSS` (example `260322`) and prevent cross-batch mixing in promotion/attendance flows.

---

## 0) Roll Format and Rules (lock before coding)

- Roll format: `YYDDSS`
  - `YY`: intake year short code from batch year (e.g., `2026 -> 26`)
  - `DD`: 2-digit department roll code (e.g., `03` for Computer)
  - `SS`: 2-digit sequence within that batch+department (`01..99`)
- Roll number is generated by backend and read-only in UI.
- Student create flow: select `batch` -> select `department` -> preview generated `roll_no` -> enter profile fields.

Decision check:
- [ ] Confirm if `SS` should be 2-digit (`01..99`) or 3-digit (`001..999`) for future scaling.

## 2) Data Backfill Strategy (existing students)

### Source assumptions
- Existing roll looks like `YYDDSS` (e.g., `780322`, `260322`).

### Backfill steps
- [ ] Parse student `roll_no` into:
  - `yy = roll_no[:2]`
  - `dd = roll_no[2:4]`
  - `ss = roll_no[4:]`
- [ ] Resolve full year (`yy`) policy:
  - If historical Nepali-style years exist, define explicit mapping table.
  - If Gregorian-only going forward, map `26 -> 2026`, etc.
- [ ] Map `dd` to `Department.roll_code` and validate exact match.
- [ ] Create/find `Batch(department, year)`.
- [ ] Set `CustomUser.batch`, `CustomUser.batch_sequence`.
- [ ] Log and skip invalid rows for manual fix.

### Backfill safety
- [ ] Dry-run first (report only).
- [ ] Hard-fail migration if duplicates in `(batch, batch_sequence)` are found.
- [ ] Export pre-migration backup of users table.
